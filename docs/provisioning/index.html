Provisioning Architecture

Test VMs boot from OCI containerDisk images stored in the local Zot registry (192.168.1.102:30500). Each pipeline run pulls the disk image and launches a fresh KubeVirt VM. The VM is torn down on exit. This replaces the older golden-disk hostDisk workflow that relied on btrfs reflink copies on the ghost node.

  • No host-side btrfs reflink: The hypervisor does not keep a raw golden disk tree that is reflink-copied per VM. Disks are distributed as compressed OCI layers and unpacked by containerd at runtime.
  • Guest filesystems vary by variant: Bluefin non-LTS uses btrfs; Bluefin LTS uses xfs; Flatcar uses ext4. The filesystem is chosen by the upstream image, not by the provisioning layer.
  • VMs float across nodes: ContainerDisk VMs schedule on any KubeVirt-capable node. PVC-backed lanes (Knuckle) co-schedule on the node that owns the PVC.
  • SSH keys are injected at runtime: KubeVirt accessCredentials propagate the public key via the QEMU guest agent after boot. No key material is baked into the published disk image.

Data Integrity Posture

  • Node capacity, utilization, and containerDisk build counts are read fromdocs/data/factory-stats.json.
  • ContainerDisk availability, digest, size, and creation time are queried live from the local Zot registry at build time; when ghost is unreachable the page falls back to the catalog definition and marks the disk unavailable.
  • Guest filesystem labels are derived from the bootc install-to-disk logic in argo/workflow-templates/build-containerdisk.yaml and are not inferred from disk names.
