QA Pipeline Architecture

Platform Verification & Quality Assurance

Bluefin is validated as an image-based, atomic operating system. Rather than verifying single packages in isolation, our testing framework asserts the entire system state, ensuring immutability, update transitions, and seamless desktop environment integrations.

01
Execution Substrate

Jobs schedule on the homelab control-plane node (ghost). The pipeline provisions **ephemeral KubeVirt VMs** for full-virtualization tests or boots **parallel Wayland test containers** running Mutter with software LLVMpipe rendering.

02
BDD & Desktop Simulation

Automated test runs trigger Python Behave BDD scenarios using **dogtail** to traverse the GNOME AT-SPI accessibility tree. This simulates real mouse moves, clicks, app launches, and desktop key combos.

03
Immutability & Integrity

The suites prove critical OS-level contracts: read-only /usr directories, writable /var paths, OCI image signatures, composefs integrity, flatpak portal boundaries, and rootless Podman execution.

The 5 Core Test Suites

smoke
GNOME Shell & Apps

Verifies desktop startup, system extension configs, top-bar menus, and primary app wrappers (Ptyxis, Software Center).

common
Sandbox & Portals

Asserts Flatpak sandbox portals, permission prompts, authorization overrides (polkit), and user space isolation boundaries.

developer
Dev Toolchains

Validates Homebrew path setup, rootless Podman socket mappings, and developer command line tool configurations.

software
Bazaar Flatpaks

Tests the desktop application centers (Bazaar hub), Flatpak CLI actions, and local app metadata caching.

system
Atomic OS Contract

Ensures bootc status OCI transport validity, composefs signatures, staged upgrade queues, and rollbacks.

How to Reproduce & Run Tests Locally

Bluefin operates as an atomic, image-based OS. You can replicate the exact dashboard BDD verification checks locally using the following steps:

  1. Verify Cluster Access: Run just list-vms to confirm connection and schedule availability on ghost control plane.
  2. Initialize Local SSH Credentials: Execute just setup-ssh-secret to build the ed25519 key-pair used by workflow pods to ssh into the test VMs.
  3. Prepare Golden Disk: Run just ensure-disk testing or just ensure-disk lts-testing to build the VM disk container.
  4. Submit Test Suite: Run just run-tests-tag testing to provision the ephemeral VM and watch the live test logs.

Core BDD Test Suites & Assertions

Each test lane executes targeted BDD scenarios written in Gherkin and backed by python-behave, dogtail, and GNOME AT-SPI tree traversal:

SuiteRuntimeKey System Assertions & Scenarios Verified
smokeContainerGNOME Shell Desktop Environment: Verifies AT-SPI registry daemon readiness, Quick Settings menus, Clock and Calendar top-bar responsiveness via Shell.Eval, Caffeine extension state, and Dash to Dock layout integrity.
commonContainerOS Integrations: Asserts flatpak sandbox portal boundaries, polkit local user authorization overrides, basic immutable host security policies, and standard desktop session persistence.
developerContainerDeveloper Toolchains: Confirms Homebrew prefix pathing, Podman rootless socket readiness, Ptyxis terminal tab multiplexing, and development build tools accessibility.
softwareContainerUser Space Application Layer: Validates user-facing software hubs, Flatpak remote sync performance, and software center API contracts.
systemKubeVirt VMAtomic OS Guarantees: Proves the core platform contract. Asserts that /usr is read-only, /var is writable, bootc status maps to a correct registry container image reference, bootc upgrade stages updates cleanly without host disruption, and composefs signatures match policies.
